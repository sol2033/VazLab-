# План разработки: ВАЗ Диагностика + Прошивка

---

## Общая концепция

Проект разрабатывается в два больших этапа:

**Этап A — Диагностика** (приоритет): кроссплатформенное приложение для телефона и десктопа, которое подключается к ЭБУ через ELM327 и показывает живые параметры, ошибки и аналитику. Жизнеспособный продукт сам по себе.

**Этап B — Прошивка**: десктопный модуль поверх той же программы для чтения, редактирования и записи прошивки через K-Line адаптер. Требует отдельного оборудования и более глубокой разработки.

---

# ЭТАП A: Диагностика

## Фаза 0: Подготовка
### ~2 недели

**0.1 Оборудование**
- ELM327 адаптер (USB или Bluetooth) — 300–800 руб.
- Установить Torque (Android) или OBD Auto Doctor — поиграться с живыми параметрами, понять как выглядят PID-запросы

**0.2 Выбор технологии для кроссплатформы**

Для одного разработчика с вайб-кодингом оптимально:
- **Flutter** — один код, работает на Android / iOS / Windows / macOS / Linux
- **Python (Kivy или BeeWare)** — альтернатива если Flutter кажется сложным
- Рекомендация: Flutter, потому что Claude и другие LLM хорошо его пишут, и UI получается нативным на всех платформах

**0.3 Изучить базу**
- Прочитать про протокол OBD-II Mode 01 (стандартные PID)
- Найти список проприетарных PID для Январь 7.2 и Bosch M7.9.7 на chiptuner.ru и forum.autoscan.ru
- Зарегистрироваться на drive2.ru — там основная аудитория будущих пользователей

## Фаза 1: Подключение и базовые параметры
### ~4 недели

**Цель:** приложение подключается к ELM327 и показывает живые данные двигателя.

**1.1 Модуль подключения к адаптеру**
- Поддержка ELM327 USB (виртуальный COM-порт)
- Поддержка ELM327 Bluetooth (на Android — через bluetooth_serial, на десктопе — виртуальный COM)
- UI: список доступных портов / Bluetooth-устройств с автоопределением
- Тест соединения: отправить AT команду, получить ответ, показать версию адаптера

**1.2 Стандартные OBD-II параметры (Mode 01)**

Список параметров для первой версии:
```
Обороты двигателя (RPM)               — PID 0x0C
Скорость автомобиля (км/ч)            — PID 0x0D
Температура охлаждающей жидкости      — PID 0x05
Положение дроссельной заслонки (%)    — PID 0x11
Напряжение бортовой сети (В)          — PID 0x42
Нагрузка на двигатель (%)             — PID 0x04
Угол опережения зажигания             — PID 0x0E
Массовый расход воздуха (ДМРВ)        — PID 0x10
Показания лямбда-зонда                — PID 0x14
Температура воздуха на впуске         — PID 0x0F
```

**1.3 Расширенные параметры ВАЗ**

Проприетарные PID для Январь 7.2 / Bosch M7.9.7:
```
Адаптивная добавка топлива (мультипликативная)
Адаптивная добавка топлива (аддитивная)
Положение регулятора холостого хода (РХХ)
Напряжение датчика детонации
Признак детонации (есть / нет)
Режим работы (закрытый / открытый контур)
```

**1.4 Дашборд реального времени**
- Набор карточек с параметрами, обновление каждые 300–500 мс
- Адаптивная вёрстка: на телефоне — вертикальный скролл, на десктопе — сетка
- Цветовая индикация: зелёный (норма) / жёлтый (внимание) / красный (критично)
- Пороговые значения нормы зашиты в базу для популярных моторов ВАЗ (21083, 21114, 21124, 21179)

## Фаза 2: Ошибки и сброс
### ~3 недели

**Цель:** полноценная работа с кодами неисправностей.

**2.1 Чтение ошибок**
- Активные ошибки (Mode 03) — список кодов
- Ожидающие ошибки (Mode 07) — коды которые появились, но ещё не активны
- Для каждого кода: числовое обозначение (P0171) + описание на русском языке
- База описаний: стандартные P0xxx — универсальная, ВАЗ-специфичные — собрать из открытых источников

**2.2 Сброс ошибок**
- Кнопка сброса (Mode 04)
- Обязательное предупреждение перед сбросом: "Убедитесь что неисправность устранена. После сброса ошибок контрольная лампа погаснет, но проблема может вернуться."
- Повторное чтение после сброса для подтверждения

**2.3 AI-объяснение ошибок**
- Кнопка "Что это значит?" рядом с каждым кодом
- Запрос к LLM (DeepSeek API): передаём код ошибки + модель двигателя
- Ответ в формате: что это, типичные причины, что проверить в первую очередь
- Кэшировать ответы локально — одинаковые коды не должны каждый раз делать API-запрос

## Фаза 3: Логирование и графики
### ~3 недели

**Цель:** запись параметров во время поездки и анализ данных.

**3.1 Запись лога**
- Кнопки "Начать запись" / "Стоп" прямо на дашборде
- Сохранение в CSV: timestamp + все активные параметры
- Имя файла: `log_ГГГГ-ММ-ДД_ЧЧ-ММ.csv`
- Список сохранённых логов с датой, длительностью и размером

**3.2 Просмотр логов**
- Графики параметров по времени
- Возможность выбрать какие параметры показывать (чекбоксы)
- Маркеры событий: резкий набор оборотов, высокая температура, детонация
- Масштабирование и прокрутка графика

**3.3 Автоматический анализ лога**
- После записи — краткая сводка: максимальные значения, выходы за пределы нормы
- AI-анализ: передать сводку в LLM, получить комментарий "На 3-й минуте поездки зафиксирована детонация при оборотах 4200. Рекомендуется проверить качество топлива или угол зажигания."

## Фаза 4: Мини-тесты и диагностические сценарии
### ~2 недели

**Цель:** автоматические тесты которые упрощают диагностику новичку.

**4.1 Тест лямбда-зонда**
- Наблюдение за переключениями в течение 30 секунд на прогретом двигателе
- Подсчёт частоты переключений (норма: 0.5–2 Гц)
- Вердикт: "Зонд исправен" / "Зонд ленивый — возможно требует замены" / "Зонд не переключается — неисправен или открытый контур"

**4.2 Тест ДМРВ**
- Считываем показания на холостом ходу на прогретом двигателе
- Сравниваем с базой норм: для 21083 — 8–9 г/с, для 21114 — 9–10 г/с, для 21124 — 10–11 г/с
- Вердикт с рекомендацией

**4.3 Тест адаптации**
- Считываем мультипликативную и аддитивную поправки
- Норма: мультипликативная ±5%, аддитивная ±5 мс
- Если за пределами нормы — AI объясняет возможные причины (подсос воздуха, засорённые форсунки, неисправный ДМРВ)

**4.4 Общий экспресс-тест**
- Кнопка "Проверить автомобиль" — запускает все тесты по очереди
- Итоговый отчёт: список проблем и рекомендаций
- Экспорт отчёта в PDF или текст для отправки мастеру

# ЭТАП B: Прошивка

> Разработка начинается после выпуска рабочей версии диагностики.
> Модуль прошивки встраивается в то же приложение, но доступен только на десктопе.

---

## Фаза 6: Подготовка к прошивке
### ~2 недели

**6.1 Дополнительное оборудование**
- ЭБУ Январь 7.2 или Bosch M7.9.7 на Авито — 500–1500 руб.
- K-Line USB адаптер для прошивки (не ELM327) — 800–1500 руб.
- Блок питания 12В / 3А — 500–800 руб.
- Arduino Uno/Nano — 300–500 руб.
- Логический анализатор USB — 400–600 руб.

**6.2 Первый эксперимент**
- Подключить ЭБУ к питанию, прочитать прошивку через Combiloader
- Открыть .bin в HxD, найти таблицу УОЗ по известному смещению
- Сохранить оригинальную прошивку — эталон на всю разработку

**6.3 Создать базу definitions**

Файл `definitions.json` — справочник смещений таблиц:
```json
{
  "I373DA02": {
    "uoz_table":  { "offset": "0x3A200", "rows": 16, "cols": 16, "units": "degrees" },
    "fuel_map":   { "offset": "0x3B100", "rows": 16, "cols": 16, "units": "ms"      },
    "idle_rpm":   { "offset": "0x1C400", "rows": 1,  "cols": 8,  "units": "rpm"     }
  }
}
```
Смещения собирать из chiptuner.ru и верифицировать руками в hex-редакторе.

## Фаза 7: Чтение и отображение прошивки
### ~4 недели

**7.1 Модуль связи K-Line**
- Реализовать протокол: инициализация, handshake, команда чтения Flash
- Тест: программа читает 64 КБ Flash и сохраняет в .bin

**7.2 Парсер прошивки**
- Читает .bin по смещениям из definitions.json
- Преобразует байты в человекочитаемые значения с учётом масштабных коэффициентов
- Поддержка нескольких идентификаторов прошивок

**7.3 UI редактора (только чтение)**
- Таблицы в виде цветной сетки (градиент по значению)
- Отображение осей (обороты, нагрузка)
- Переключение между таблицами в боковом меню
- Связка с диагностикой: подсветка активной ячейки по текущим оборотам и нагрузке

## Фаза 8: Диагностика ЭБУ перед прошивкой
### ~3 недели

**Цель:** автоматический чеклист безопасности, без прохождения которого запись заблокирована.

**8.1 Pre-flash wizard**

```
Шаг 1 — Подключение
  ✓ Адаптер обнаружен
  ✓ Связь с ЭБУ установлена
  ✓ Идентификатор прошивки найден в базе

Шаг 2 — Диагностика железа
  ✓ Напряжение бортовой сети: XX.X В  (блок если < 11.8 В)
  ✓ Стабильность соединения: 0 потерь из 100 пакетов
  ⚠ Время чтения сектора 3 повышено

Шаг 3 — Диагностика прошивки
  ✓ CRC текущей прошивки корректна
  ✓ Резервная копия сохранена: backup_I373DA02_2024-11-01.bin
  ✓ Подозрительных паттернов не обнаружено

Итог: [ПРЕДУПРЕЖДЕНИЕ / ОК / ЗАПРЕЩЕНО]
```

**8.2 Пороговые значения**

| Параметр | Норма | Предупреждение | Запрет записи |
|---|---|---|---|
| Напряжение | > 12.5 В | 11.8–12.5 В | < 11.8 В |
| Потери пакетов | 0% | 0% | > 0% |
| CRC текущей прошивки | Верна | — | Неверна |
| Идентификатор в базе | Найден | — | Не найден |

## Фаза 9: Редактор и запись прошивки
### ~5 недель

**9.1 Редактируемые таблицы**
- Ячейки таблиц становятся редактируемыми
- Изменённые ячейки подсвечиваются отдельным цветом
- Кнопка "Отменить изменения" возвращает к прочитанным значениям
- Soft-лимиты: минимальное и максимальное допустимое значение на каждый параметр

**9.2 AI-предупреждения при редактировании**

При изменении ячейки — запрос к LLM с полным контекстом:
```
"Пользователь меняет УОЗ с 18° на 24° при 3000 об/мин.
 Двигатель: 21114. Текущие данные диагностики: детонации нет,
 лямбда 0.98, температура 85°C.
 Оцени риски кратко."
```

**9.3 Пересчёт CRC**
- После любого редактирования — автоматический пересчёт контрольной суммы
- Алгоритмы CRC для поддерживаемых блоков реализуются по документации сообщества

**9.4 Запись в ЭБУ**
- Секторная запись (только изменённые секторы, не вся Flash целиком)
- Прогресс-бар с текущим сектором и процентом
- Верификация каждого сектора сразу после записи
- При несовпадении верификации — немедленная остановка с сообщением
- Кнопка "Восстановить резервную копию" доступна всегда

## Фаза 10: Тестовый стенд
### ~3 недели (параллельно с фазами 8–9)

**Цель:** тестировать программу без риска для личного автомобиля.

**10.1 Минимальный стенд (Этап 1)**
- ЭБУ + питание 12В + K-Line адаптер
- Только тестирование чтения / записи / верификации
- Датчики не нужны

**10.2 Стенд с симулятором датчиков (Этап 2)**
- Arduino генерирует сигнал ДПКВ (58 зубьев, переменная частота = обороты)
- Резисторы вместо ДТОЖ и ДМРВ
- Логический анализатор измеряет момент искры и длительность впрыска
- Проверка: изменил УОЗ в таблице → измерил угол → совпадает с настройкой

**10.3 Чеклист приёмочных тестов**
- 30 циклов чтение-запись-верификация без единой ошибки
- Запись при пограничном напряжении (12.0 В) — должна блокироваться
- Попытка записать прошивку с неверным идентификатором — должна блокироваться
- Попытка записать файл с битым CRC — должна блокироваться
- Восстановление резервной копии — должно работать корректно

## Ключевые риски и как их снизить

| Риск | Как снизить |
|---|---|
| Смещения таблиц в definitions окажутся неверными | Верифицировать каждое смещение руками в HxD до реализации |
| ELM327 не поддерживает расширенные ВАЗ PID | Протестировать на реальном автомобиле до начала разработки этой части |
| Flutter сложен для новичка | Начать с простейшего экрана, активно использовать Claude для генерации кода |
| Кирпич ЭБУ при тестировании | Всегда тестировать на стендовом ЭБУ, никогда на рабочем автомобиле |
| Малый отклик аудитории | Выложить бесплатную диагностику на drive2.ru ещё в бете — собрать фидбек заранее |
